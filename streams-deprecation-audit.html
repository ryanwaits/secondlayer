Issues & Risks

Critical considerations for deprecation

1
Database Migration Required
Three tables must be handled: streams, stream_metrics, and deliveries. The jobs table has a stream_id foreign key that blocks simple deletion.
→ fix: 1) Archive deliveries to cold storage 2) Migrate active streams to workflow triggers 3) Drop tables with CASCADE
2
Breaking SDK Changes
The SecondLayer.streams client is a public API. Removing it breaks all SDK consumers. 166 lines of client code plus tests must be removed or deprecated gracefully.
→ fix: Add deprecation warnings in v1.x, remove in v2.0 with migration guide
3
Workflow StreamTrigger Type Exists
The workflows package has a StreamTrigger type (line 427 in workflows.ts). This creates naming confusion if Streams is deprecated but StreamTrigger remains.
→ fix: Rename StreamTrigger to EventTrigger or similar before Streams removal
4
CLI Cross-Dependencies
stack.ts references streams for dev environment orchestration (--wait flag, pause/resume on stop). setup.ts creates a streams directory and prompts for stream configuration.
→ fix: Remove stream-related flags from stack command, simplify setup flow
5
Simplify to 2-Product Model
Deprecating Streams leaves Subgraphs and Workflows as the two core products. This simplifies positioning: query via subgraphs, act via workflows. No more webhook complexity.
→ consider: Update all marketing copy, docs, and CLI setup wizard
6
Plan Limits Reference
shared/src/lib/plans.ts has a streams: number field in plan limits. The starter plan allows 3 streams. These limits become obsolete.
→ fix: Remove streams from plan limits, update account usage queries

Streams Feature Workflows Equivalent Migration Effort Notes

Delivery logs Run logs Medium Different UI, same data retention
Pause/resume stream Enable/disable workflow Low Same functionality, different API
Bulk pause/resume Not supported High Script required to toggle multiple workflows
Stream metrics (deliveries, success rate) Run analytics Medium New metrics table needed
Archive Strategy: Before deletion, archive the following for reference:
  • All deliveries data (compliance/auditing)
  • Active stream configurations (for workflow migration)
  • Stream metrics history (usage analytics)
  • Git tag before deletion (git tag streams-final)
Migration Script Idea: Create sl streams migrate-to-workflow <stream-id> that converts a stream's filters and endpoint into a workflow definition with a deliver step. This bridges the gap for active users.
