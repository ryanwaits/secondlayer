# PRD 0005 - Phase 2 Datasets, Console, and Streams Bulk Dumps

**Status:** Draft
**Owner:** Ryan

## Summary

Phase 2 turns the Phase 1 read surfaces into public artifacts and self-serve product infrastructure:

1. Stacks Datasets: five curated, public-good datasets with stable APIs, downloadable parquet, schema docs, and lightweight dashboard views.
2. Streams bulk dumps: the L1 archive path for backfills, exposed as parquet files with a manifest.
3. Console v1: enough self-serve account surface for paid tenants to manage keys, inspect usage, and see their Subgraphs and Subscriptions.

This PRD is a planning draft. It does not authorize changing public cursor semantics, dataset schemas, pricing, parquet manifest format, or billing contracts without explicit approval.

## Context

Phase 1 is closed enough to proceed: Streams and Index production smoke is green, `/public/status.services[]` reports required services as `ok`, Staging Health passed twice, and fresh logical backup/WAL evidence is recorded. Full backup/PITR proof, remote restore drill, server expansion, hot-spare capacity, and failover rehearsal remain deferred to the funded infrastructure milestone.

Phase 2 should reuse the existing data plane rather than creating parallel ingestion paths. Streams bulk dumps come from L1 canonical events. Stacks Datasets come from L2 decoded tables or dataset-specific materialized views. Console v1 reads from the account, key, usage, billing, subgraph, and subscription state already owned by the platform.

## Goals

1. Ship the first five Stacks Datasets as public, queryable, documented data products.
2. Ship Streams bulk dumps for historical L1 backfills.
3. Give developers a minimal Console v1 for keys, usage, billing state, subgraphs, and subscriptions.
4. Make the dataset and dump outputs operationally boring: generated by jobs, observable, resumable, and documented.
5. Preserve the Phase 1 API contracts while adding bulk and public-good surfaces.

## Non-goals

- New raw-event push delivery or webhooks from Stacks Streams.
- Cursor format changes.
- New Stacks Index endpoint families beyond those approved by a dataset-specific decision.
- Console as a full admin/billing replacement for manual operator workflows.
- Full PITR restore drill, hot-spare failover, or second-node infrastructure.
- Dataset marketplace, rev share, or third-party dataset publishing.
- Multi-chain or non-Stacks datasets.

## Scope

### Streams bulk dumps

Ship a cold/backfill path for Stacks Streams.

Required v0:

- Canonical L1 events exported to parquet.
- Files partitioned by block-height range.
- A machine-readable manifest with:
  - dataset name and version,
  - network,
  - generated timestamp,
  - block range coverage,
  - file list with row counts and checksums,
  - schema version,
  - latest finalized cursor included.
- Repeatable generation job that can rebuild a range.
- Incremental publishing job for recent finalized ranges.
- Public docs explaining the intended workflow: sync historical parquet, then tail `GET /v1/streams/events` from a recent cursor.

Open decisions before implementation:

- Object storage provider and bucket naming.
- Public URL shape.
- Block partition size.
- Manifest schema and versioning rules.
- Reorg/finality policy for when a range is safe to publish.

### Stacks Datasets

Ship five public-good datasets. Each dataset gets:

- Stable read API.
- Parquet download path.
- Schema reference.
- Small dashboard view.
- Freshness/status signal.
- Backfill job and update job.

Initial dataset shelf:

| Dataset | Source | v0 shape |
|---|---|---|
| STX transfers | L2 decoded STX transfers | transfers by sender, recipient, block range, amount |
| PoX-4 / Stacking | decoded events plus protocol rules | cycles, delegations, pool/signer metrics |
| sBTC | SIP-010 and protocol-specific events | mints, burns, transfers, supply snapshots |
| BNS | BNS contract events | names, namespaces, ownership history, renewals where available |
| Network health | L1 blocks, reorgs, API telemetry | block times, ingest lag, decoder lag, reorg counts |

Open decisions before implementation:

- Exact schema for each dataset.
- Whether each dataset is a direct API endpoint, a named Subgraph, a materialized L2 table, or a hybrid.
- Update cadence and freshness SLO.
- Whether public parquet is free for all or gated by tier for heavy use.
- Which dashboard charts are required for launch versus follow-up.

### Console v1

Ship the minimum self-serve account surface for Build/Scale tenants.

Required v0:

- API key list, create, rotate/delete where supported.
- Usage page showing Streams and Index metering by day.
- Billing page showing current plan and trial state.
- Subgraph list with status, last processed block, row counts, and links.
- Subscription list with status, delivery health, and links.
- Account/team settings sufficient for operator-supported onboarding.

Console v1 may be partial if it is honest about what remains manual. It should not block Phase 2 datasets if the public artifact work is ready first.

Open decisions before implementation:

- Stripe checkout and portal scope for Build/Scale.
- 30-day trial entry point and conversion rules.
- Which account/team roles exist in v1.
- Which admin-only controls stay manual.

### Documentation and marketing

Required v0:

- Stacks Datasets page describing the shelf and each dataset.
- Streams bulk dumps docs linked from Stacks Streams.
- Console docs or onboarding page.
- API/SDK examples for reading datasets and switching from parquet backfill to Streams tailing.
- Status/freshness language for each public dataset.

## Product constraints

- Product names remain: Stacks Streams, Stacks Index, Stacks Subgraphs, Stacks Datasets, Console.
- Stacks Streams remains read-only.
- Stacks Datasets are public-good artifacts; heavy programmatic use can roll into paid tiers.
- Parquet and dataset schemas become public contracts once launched. Treat their versioning as irreversible until explicitly approved.

## Proposed sequence

1. Resolve public contracts: dataset schemas, parquet manifest, partitioning, and URL shape.
2. Build Streams bulk dump generator and manifest writer.
3. Build dataset generation jobs for the first dataset or two.
4. Add public dataset APIs and schema docs.
5. Add dashboard views and marketing/docs pages.
6. Fill Console v1 gaps needed for key management, usage, and billing state.
7. Run production smoke for dumps, datasets, and Console navigation.

## Acceptance criteria

1. Streams bulk dumps publish parquet files and a manifest for a known block range.
2. A backfill consumer can read parquet and resume through `GET /v1/streams/events` from the documented cursor.
3. The five Stacks Datasets have stable schema docs.
4. Each dataset has a read API and parquet path.
5. Each dataset has a basic dashboard view.
6. Dataset freshness is visible in status or dataset-specific metadata.
7. Console v1 lets a paid tenant inspect API keys, usage, plan/trial state, Subgraphs, and Subscriptions.
8. Pricing/docs copy is aligned with Build, Scale, and Enterprise access to dumps and heavy dataset usage.
9. Production smoke covers at least one parquet manifest, one dataset API, one dataset dashboard, and Console navigation.

## Validation plan

- Unit tests for manifest generation and schema serialization.
- Integration tests for dump generation over a small block range.
- API tests for each dataset endpoint.
- Web smoke tests for dataset pages, dashboard views, and Console pages.
- SDK examples typecheck if dataset helpers are added.
- Production smoke against:
  - manifest URL,
  - one parquet object checksum,
  - one dataset API,
  - one dataset dashboard,
  - Console key/usage/subgraph/subscription pages.

## Open questions

1. What is the minimum useful v0 schema for each of the five datasets?
2. Should datasets be exposed under `/v1/datasets/...`, `/v1/index/...`, or a separate public host path?
3. Which object storage provider and CDN are acceptable for public parquet?
4. What block range partition size keeps backfills ergonomic without creating too many objects?
5. Should Console v1 ship before all five datasets, or can it trail the first dataset/dump release?
6. Which Phase 2 item is the first Sunday sprint target: Streams bulk dumps, one dataset vertical slice, or Console self-serve basics?
