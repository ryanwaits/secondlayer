PoX Cycle Aggregate Design

Already shipped: /v1/index/stacking returns raw pox4_calls rows with filters on function_name, stacker, start_cycle, etc. What's missing is the cycle-level rollup — the aggregate shape every stacking dashboard hand-builds.

New routes

GET /v1/index/pox/cycles
Paginated list of reward cycles, newest first. Filters: from_cycle, to_cycle, limit, cursor.
GET /v1/index/pox/cycles/:cycle
Single reward cycle detail. Returns the same aggregate shape as the list item but with extended signer breakdown.
Cache policy
Completed cycles → long-lived cache (immutable once cycle closes). Current cycle → short cache (5–30s, still accumulating).
Source table
pox4_calls — GROUP BY reward_cycle with canonical = true filter.
New file
packages/api/src/index/pox-cycles.ts — mirrors the sbtc-peg.ts pattern exactly.

PoxCycle response shape

reward_cycle
number — cycle index (1, 2, … current)
total_stacked_ustx
string (bigint) — sum of amount_ustx across stack-* calls
unique_stackers
number — COUNT DISTINCT stacker
unique_delegators
number — COUNT DISTINCT caller on delegate-* calls
action_count
number — total rows in the cycle
start_block_height
number — MIN block_height in cycle
end_block_height
number | null — MAX (null if current cycle)
function_breakdown
Record of function_name → count (e.g. {"stack-stx": 12, "delegate-stx": 45})

Risks & Design Notes

1
sbtc_events is on the target DB, trigger-evaluator must use the right DB handle
The DB split means sbtc_events (chain data) lives on TARGET_DATABASE_URL, same as decoded_events. The trigger-evaluator already uses the target DB for chain events — just use the same handle for the new sbtc query. Don't accidentally use the control-plane DB handle.
→ use targetDb / same handle as decoded_events queries; don't introduce a new connection
2
PoX cycle boundary: how to detect "current cycle" vs completed
Completed cycles have a fixed end_block_height; the current cycle is still accumulating. Cache policy should be: reward_cycle < current_cycle → long TTL (immutable); reward_cycle === current_cycle → short TTL (30s). Need to know the current cycle number — derive it from MAX(reward_cycle) in the query or pass chain tip to the resolver.
→ SELECT MAX(reward_cycle) FROM pox4_calls WHERE canonical as part of the cycles response to flag the open cycle
3
sbtc_withdrawal_accept carries sweep_txid but sweep confirmation is NOT yet verified
The ROADMAP notes that withdrawal-accept means the signers committed to broadcast sweep_txid, not that the BTC tx confirmed. The webhook payload should include sweep_txid and a clear settlement_confirmed: false flag — don't imply BTC finality until the BTC confirmer ships.
→ add settlement_confirmed: boolean field to SbtcWithdrawalEvent; hardcode false until BTC confirmer builds
4
PoX function_breakdown: serialize as object or typed array?
A Record<string, number> is easy to produce from jsonb_object_agg in Postgres but awkward to type in OpenAPI. A typed array {function_name, count}[] is more explicit and easier to document. Given the 12 known function names are a finite enum, a typed array is the right call.
→ use { function_name: Pox4FunctionName; count: number }[] array; reuse the existing SUPPORTED_FUNCTIONS union from the decoder
5
Subscription filter matching: amount filters need bigint comparison
sBTC amounts are in satoshis (bigint territory). The min_amount / max_amount filter fields in the subscription schema need to handle numeric strings, not JS numbers, to avoid precision loss. The existing stx_transfer minAmount filter is a useful precedent — check how it handles bigint comparison in the evaluator.
→ store amounts as string in the Zod schema; compare with BigInt() in the matcher; mirror stx_transfer minAmount handling exactly
6
PoX cycles route collision with existing /stacking route
The existing raw-calls route lives at /v1/index/stacking. New cycle aggregates should go at /v1/index/pox/cycles — a different path segment. Double-check routes/index.ts for any existing /pox prefix that might conflict before registering.
→ register under /pox/cycles and /pox/cycles/:cycle; the existing /stacking path stays unchanged

Recommended Sprint Order

Sprint 1 — sBTC webhook schema (shared package): Add 4 Zod union variants to subscriptions.ts, add SbtcDepositEvent / SbtcWithdrawalEvent to chain-envelopes.ts. Zero runtime risk — schema changes only. Validates: CHAIN_TRIGGER_FIELDS auto-derives correct filter keys for each topic.
Sprint 2 — Trigger evaluator extension (subgraphs package): Add loadSbtcEventsForBlock() and the second match loop in evaluateBlock(). Validates: write an integration test that inserts a fake sbtc_events row + a matching subscription and confirms an outbox row is produced.
Sprint 3 — PoX cycle aggregates (api package): New pox-cycles.ts module + 2 routes registered in routes/index.ts. Validates: HTTP handler test hitting real pox4_calls data; check function_breakdown array, pagination cursor, cache headers.
Changesets: @secondlayer/shared (minor — new topic types), @secondlayer/subgraphs (minor — new evaluator behavior), @secondlayer/api (minor — new routes). Publish all three together after Sprint 3 green.
