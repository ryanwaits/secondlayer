Recommendation summary. Drop /billing from the sidebar. Rename the Instance entry to Instance (unchanged label, broader scope). Keep /platform/billing as a server redirect to /platform/instance?upgrade=success#plan so the Stripe webhook return URL keeps working without code changes upstream. Usage analytics stays at /usage.

Mobile-specific behavior

  • Right rail anchors collapse into a horizontal scrolling tab strip pinned below the topbar (Overview · Plan · Spend · Payment · Access · Danger). Treat sections as discrete views, not anchor-jumps.
  • Plan table rows reflow vertical: each plan becomes a stacked card (name · price on row 1; vCPU/RAM/Storage on row 2; notes + action on row 3). Still no card-in-card — the plan list itself is the only card.
  • Health grid goes from 4 cells in a row to a 2×2 grid. Container list collapses to a single line: 3/3 healthy · postgres · api · indexer.
  • KV rows (Overview / Access) reflow: key on its own line above the value. Action buttons drop to the right of the value or below if they overflow.
  • Spend form stacks vertically (cap, then threshold).
  • Confirm-resize banner stacks the buttons vertically and increases tap target to ~44px.
06 What happens to the old Billing sidebar item

Remove from sidebar pick

Drop the Billing entry under Settings entirely. The unified Instance page now owns plan, spend, and payment.

Sidebar gains zero new entries; it just loses one.

Redirect /platform/billing pick

Server redirect to /platform/instance#payment, preserving query params. This protects the Stripe Checkout return URL (?upgrade=success) and any in-flight email links.

Old route stays addressable for ~3 months, then we drop the file.

Rename to "Instance & Billing" skip

Reads as a marketing label, not an operations one. The page is fundamentally about the instance — billing is a tab inside it.

Keep the label simply Instance.
