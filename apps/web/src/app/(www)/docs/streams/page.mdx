# Streams

Raw chain event firehose ‚Äî cursor-paginated REST, idempotent, replayable.

Streams expose the same event surface every Foundation Dataset decoder consumes internally. Page forward with a cursor, replay from any point, and never miss an event. Stream live over SSE in-process (see below), or fan out to your backend with [Subscriptions](/docs/subscriptions).

Streams reads require an API key ‚Äî send it as `Authorization: Bearer`.

## Read the stream

```bash
curl -H "Authorization: Bearer $SL_API_KEY" \
  "https://api.secondlayer.tools/v1/streams/events?limit=100"
```

Each response returns a `next_cursor`. Pass it back as `cursor` to continue exactly where you left off ‚Äî deliveries are idempotent, so retries never double-count.

```bash
curl -H "Authorization: Bearer $SL_API_KEY" \
  "https://api.secondlayer.tools/v1/streams/events?cursor=7970329:8"
```

## Filter the stream

Narrow the firehose by event type, contract, or principal ‚Äî `types`, `contract_id`, `sender`, `recipient`, `asset_identifier`, and `from_height`/`to_height`. Filters are exact-match; event types without a given field simply don't match.

```bash
curl -H "Authorization: Bearer $SL_API_KEY" \
  "https://api.secondlayer.tools/v1/streams/events?types=ft_transfer&sender=SP...&limit=50"
```

## Real-time firehose (SSE)

`GET /v1/streams/events/stream` streams events as Server-Sent Events over a single long-lived connection ‚Äî the native push surface inside Streams. Authenticate with `Authorization: Bearer`, and apply the same filters as `/events`: `types`, `not_types`, `contract_id`, `sender`, `recipient`, and `asset_identifier`.

```bash
curl -N -H "Authorization: Bearer $SL_API_KEY" \
  "https://api.secondlayer.tools/v1/streams/events/stream?types=ft_transfer"
```

Choose a start position with `cursor` (or `from_cursor`), or `from_height`. With no start position the connection live-tails from the reorg-clamped tip. A `ping` keepalive frame is sent every 20s so idle connections stay open.

The SDK's `client.events.subscribe(...)` is the ergonomic way to consume this ‚Äî it carries your Bearer key, auto-reconnects from the last delivered cursor, and verifies per-frame signatures. See the [SDK docs](/docs/sdk).

### Per-frame signatures

Each SSE frame body is `{ event, sig, key_id }`, where `sig` is an ed25519 signature over `JSON.stringify(event)`. When the Streams client is created with `verify`, every frame is checked against the key published at `/public/streams/signing-key` before your handler runs ‚Äî a bad or missing signature throws. This is the streaming counterpart to the signed `X-Signature` on REST reads: trust the data without trusting the server.

## What's new (May 2026)

- **Finality** ‚Äî `tip.finalized_height` plus a `finalized` flag on every event, anchored to Bitcoin (burn-block) confirmations. A `finalized` event won't reorg.
- **Caching** ‚Äî fully-finalized pages are served `Cache-Control: public, immutable` with an `ETag` (and `304` on `If-None-Match`); the moving tip stays short-lived.
- **Signed responses** ‚Äî every read carries an ed25519 `X-Signature`. Fetch the public key at `/public/streams/signing-key` and verify, or set `verify` on the SDK client ‚Äî trust the data without trusting the server.
- **Bulk + replay** ‚Äî download finalized history as parquet (`sl streams pull`, or the SDK `dumps` namespace) and `events.replay({ from: "genesis" })` to backfill cold history then tail live with no gap at the seam. The bulk manifest is itself ed25519-signed and is verified before any per-file `sha256` is trusted ‚Äî the manifest signature is the root of trust, with the file hashes chained beneath it. The client option `verifyDumpsManifest` defaults to `true` (both `dumps.list()` and `events.replay()` enforce it); opt out with `verifyDumpsManifest: false`.
