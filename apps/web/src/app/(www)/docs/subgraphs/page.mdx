# Subgraphs

Subgraphs are your schema on our indexer: one TypeScript file declares the contract events you care about and a handler that shapes each event into rows, and a deploy gives you hosted Postgres tables behind the same public `/v1` read API as Index. Secondlayer backfills history and keeps the tables live as new blocks arrive. Deploying a subgraph requires a paid plan or a 14-day trial â€” a free account gets `403 PLAN_REQUIRED`.

## How it works

- **Declare sources** â€” the contracts and events you want to track.
- **Write a handler** â€” turn each matched event into one or more rows.
- **Deploy** â€” a paid plan or 14-day trial is required (free accounts get `403 PLAN_REQUIRED`); Secondlayer backfills history from `startBlock`, then streams new blocks into your table in real time.

## Define a subgraph

A subgraph is a single config file: named source filters, a table schema, and handlers keyed by source name. `startBlock` is top-level on the definition:

```ts
import { defineSubgraph } from "@secondlayer/subgraphs";

export default defineSubgraph({
  name: "sbtc-flows",
  startBlock: 8000000,
  sources: {
    transfers: {
      type: "ft_transfer",
      assetIdentifier: "SM3VDXK3WZZSA84XXFKAFAF15NNZX32CTSG82JFQ4.sbtc-token::sbtc-token",
    },
  },
  schema: {
    transfers: {
      columns: {
        tx_id: { type: "string" },
        block_height: { type: "uint" },
        amount: { type: "uint" },
        sender: { type: "string" },
        recipient: { type: "string" },
      },
    },
  },
  handlers: {
    transfers: (event, ctx) => {
      ctx.insert("transfers", {
        tx_id: ctx.tx.txId,
        block_height: ctx.block.height,
        amount: event.amount,
        sender: event.sender,
        recipient: event.recipient,
      });
    },
  },
});
```

## Write a handler

Handlers are values in the definition's `handlers` map, keyed by source name (or `"*"` for a catch-all). Event payloads are flat and typed per source (`event.amount`, `event.sender` â€” no wrapper object); tx and block metadata come from `ctx.tx` and `ctx.block`, and writes go through the ctx mutators (`ctx.insert`/`update`/`upsert`/`delete`/`patch`). Every row your handler writes lands in the materialized table.

```ts
import { defineSubgraph } from "@secondlayer/subgraphs";

export default defineSubgraph({
  name: "sbtc-flows",
  startBlock: 8000000,
  sources: {
    transfers: {
      type: "ft_transfer",
      assetIdentifier: "SM3VDXK3WZZSA84XXFKAFAF15NNZX32CTSG82JFQ4.sbtc-token::sbtc-token",
    },
  },
  schema: { /* transfers table */ },
  handlers: {
    // key matches the source name; each matched event becomes one row
    transfers: (event, ctx) => {
      ctx.insert("transfers", {
        tx_id: ctx.tx.txId,
        block_height: ctx.block.height,
        amount: event.amount,
        sender: event.sender,
        recipient: event.recipient,
      });
    },
  },
});
```

## Typed print payloads

Clarity ABIs don't describe `print` payloads, so `event.data` on a `print_event` source is untyped by default. Declare a `prints` map â€” per-topic field â†’ column type â€” and `event.data` becomes fully typed, discriminated on `event.topic`:

```ts
sources: {
  registry: {
    type: "print_event",
    contractId: "SM3VDXK3WZZSA84XXFKAFAF15NNZX32CTSG82JFQ4.sbtc-registry",
    topic: "completed-deposit",
    prints: {
      "completed-deposit": { bitcoinTxid: "text", amount: "uint" },
    },
  },
},
handlers: {
  registry: (event, ctx) => {
    event.data.bitcoinTxid; // string â€” typed, no casting
    event.data.amount;      // bigint
  },
},
```

Don't guess the fields â€” payload shape varies per topic within one contract, and a guessed field silently nulls forever. Scaffold the whole definition from the contract's **observed** history instead:

```bash
sl subgraphs create sbtc-registry \
  --from-contract SM3VDXK3WZZSA84XXFKAFAF15NNZX32CTSG82JFQ4.sbtc-registry
```

This calls the [print-schema endpoint](/docs/index#print-schemas) â€” per-topic schemas inferred empirically from indexed print events â€” and writes one `print_event` source per topic with its `prints` map, plus a single wide table with per-column "null except on topics: â€¦" comments (`--table-per-topic` for a table per topic). No login needed. `sl subgraphs codegen <file> --payloads` emits a `.d.ts` of the per-topic payload types next to the definition.

As a safety net, deploys return advisory `warnings` when a handler reads an `event.data` field never observed on-chain for its source's topic (pinned `contractId` sources only â€” wildcard and trait sources are skipped). Warn-only, never blocks; the CLI prints them.

## Deploy

Deploy from the CLI. Deploying requires a paid plan or a 14-day trial â€” a free account gets `403 PLAN_REQUIRED` and must start a trial or plan first. On a plan or trial, the first deploy backfills history from `startBlock`; subsequent blocks stream in automatically.

**Tip-first deploys.** Add `--tip-first` (or `backfillMode: "concurrent"` in the definition) and your subgraph goes live at the chain tip immediately â€” rows are queryable in seconds while history backfills in the background. Only use it when your handlers tolerate out-of-order processing (commutative writes like balances and counters, or insert-only tables); "latest-value-wins" handlers should keep the default blocking backfill.

**No account?** When the x402 pay-per-call rail is enabled, `POST /v1/subgraphs` accepts an x402 payment ($2) in place of an API key: the subgraph is owned by the paying wallet, indexes forward from deploy, and expires after 7 days unless renewed (`POST /v1/subgraphs/{name}/renew`, $0.50) or the account is claimed.

```bash
sl subgraphs deploy ./subgraph.config.ts
```

## Visibility & publishing

Every subgraph has a **visibility**. Managed deploys default **public** â€” anon-readable on `/v1`. Deploys to your own database (`--database-url`) default **private**, and existing subgraphs were migrated private.

Override at deploy, or flip later:

```bash
sl subgraphs deploy ./subgraph.config.ts --visibility private
sl subgraphs publish sbtc-flows     # make public
sl subgraphs unpublish sbtc-flows   # make private again
```

The SDK exposes the same as `sl.subgraphs.publish(name)` / `unpublish(name)`, and MCP as `subgraphs_publish` / `subgraphs_unpublish` (plus a `visibility` param on `subgraphs_deploy`).

Public names are a **single global namespace**, claimed on publish â€” a taken name returns `409 PUBLIC_NAME_TAKEN`. Private subgraphs stay readable on `/v1` with your `sk-sl_` bearer; anonymous requests to a private name return `404` (no existence leak).

## Query your data

Your table is immediately available over REST at `/v1/subgraphs` â€” anon-readable for public subgraphs (managed deploys default public), wildcard CORS. Pages are `_id`-keyset: pass `?cursor=<next_cursor>` to resume, `_order=asc|desc` for direction. No `_offset`/`_sort` on `/v1`.

```bash
curl https://api.secondlayer.tools/v1/subgraphs/sbtc-flows/transfers \
  -G -d "_limit=10" -d "_order=desc"
```

```json
{
  "rows": [
    {
      "_id": 48138,
      "block_height": 8054704,
      "tx_id": "0x5ee90c4f8a21d25b",
      "amount": 697078828,
      "sender": "SP3PE7Q9...X44J"
    }
  ],
  "next_cursor": "48137",
  "tip": { "block_height": 8054704, "subgraph_height": 8054704, "blocks_behind": 0 }
}
```

`GET /v1/subgraphs` lists public subgraphs (plus your own with a bearer), and `GET /v1/subgraphs/sbtc-flows` returns metadata â€” tables, columns, and the indexing tip. Row-by-id and counts live at `/:table/:id` and `/:table/count`.

## Aggregate without pulling rows

Roll up a table server-side instead of paging every row: `count`, `sum`, `min`, `max`, and distinct counts over the same filters as the list endpoint.

```bash
curl https://api.secondlayer.tools/v1/subgraphs/sbtc-flows/transfers/aggregate \
  -G -d "_count=true" -d "_sum=amount" -d "_countDistinct=sender"
```

Anon for public subgraphs; private subgraphs need the owning account's `sk-sl_` bearer (anon gets `404`).

`_sum`/`_min`/`_max` take numeric columns only. The typed SDK exposes the same as `<table>.aggregate(spec)` â€” see [SDK aggregates](/docs/sdk#aggregate-subgraph-rows) and the [REST API reference](/docs/rest-api#aggregates) for the full parameter set.

## Index by standard, not by address

A source can target a token standard instead of a fixed contract. `trait` resolves to **every contract Secondlayer classifies as that standard** â€” including ones deployed after you ship the subgraph â€” so you index a whole ecosystem with no address list to maintain:

```ts
sources: {
  // every SIP-010 token transfer on-chain
  tokens: { type: "ft_transfer", trait: "sip-010" },
},
```

Supported traits: `sip-009`, `sip-010`, `sip-013`. On paid plans, a reindex backfills each contract's full history from its deploy block, even if it was classified later (without a plan, reindexes are clamped to the subgraph's registered start). See [Contract discovery](/docs/contracts) to query the matching set directly.

## Stream rows in real time

Tail a table over Server-Sent Events as rows are indexed, instead of polling. The endpoint streams from the current tip by default, or pass `?since=<block_height>` to replay from a height and then tail live. Column filters are accepted as query params, and a periodic `ping` event keeps the connection alive.

```bash
curl -N https://api.secondlayer.tools/v1/subgraphs/sbtc-flows/transfers/stream \
  -G -d "since=8050000"
```

Anon for public subgraphs; private ones need the owning account's `sk-sl_` bearer.

From the SDK, `subscribe` wraps the stream with the same `where` filters as `findMany` and returns an unsubscribe function:

```ts
const unsubscribe = subgraph.transfers.subscribe(
  (row) => console.log(row),
  { where: { sender: "SP3PE7Q9...X44J" }, since: 8050000, onError: console.error },
);
```

`subscribe` uses the global `EventSource`, so it runs in the browser or **Node â‰Ą 22** only.

## Use your own database + ORM

Deploy with `--database-url` and your subgraph's rows land in **your** Postgres. Then generate a typed schema for your ORM and query it with full joins and relations:

```bash
sl subgraphs deploy ./subgraph.config.ts --database-url "$DATABASE_URL"
sl subgraphs codegen ./subgraph.config.ts --target prisma -o prisma/schema.prisma
npx prisma generate
```

Prisma, Drizzle, and Kysely all have first-class generators (`--target prisma|drizzle|kysely`). Secondlayer owns the table DDL, so treat the tables as read-only and verify with `prisma db pull` (never `migrate`).

Want pushes instead of polling? Bind a [Subscription](/docs/subscriptions) to this table and Secondlayer will POST every matching row change to your webhook â€” inserts, updates, and deletes, with the verb encoded in the payload `type` (see the [Subscriptions](/docs/subscriptions) page).

### Breaking changes on your own database

For managed subgraphs a breaking schema change (removed table/column, changed column type, or a forced reindex) auto-reindexes by dropping and rebuilding the schema. On a BYO database that drop would destroy your data, so the deploy is **refused** (HTTP `422`) â€” nothing on your database is touched.

The refusal carries the exact migration plan instead of a generic error. `sl subgraphs deploy` prints the breaking reasons and the `DROP SCHEMA â€¦ CASCADE` + rebuild DDL to run yourself, then re-deploy:

```bash
âś— Refusing breaking schema change on BYO subgraph (no data dropped).
Breaking changes:
  âś— transfers: removed columns [amount]
To rebuild manually, run on YOUR database:
  DROP SCHEMA IF EXISTS "sg_my_subgraph" CASCADE;
  CREATE SCHEMA "sg_my_subgraph";
  CREATE TABLE â€¦
```

The SDK throws a typed `ByoBreakingChangeError` exposing `details.reasons` and `details.plan` (`dropStatement`, `statements`, `grantScript`) for programmatic handling. A destructive `--force` rebuild on your database is **not yet supported** â€” run the DROP manually.
