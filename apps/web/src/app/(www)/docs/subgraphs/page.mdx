# Subgraphs

Stacks Subgraphs are the L3 surface for app-specific materialized tables. Declare the contract events you care about, write a handler that shapes each event into rows, and Secondlayer backfills history and keeps the table live as new blocks arrive.

## How it works

- **Declare sources** тАФ the contracts and events you want to track.
- **Write a handler** тАФ turn each matched event into one or more rows.
- **Deploy** тАФ Secondlayer backfills history, then streams new blocks into your table in real time.

## Define a subgraph

A subgraph is a single config file. Point it at one or more contract sources and the events you care about:

```ts
import { defineSubgraph } from "@secondlayer/subgraphs";

export default defineSubgraph({
  name: "sbtc-flows",
  network: "mainnet",
  sources: [
    {
      contract: "SM3VDXK3WZZSA84XXFKAFAF15NNZX32CTSG82JFQ4.sbtc-token",
      events: ["ft_transfer_event"],
      startBlock: 8000000,
    },
  ],
});
```

## Write a handler

Handlers are typed by their source: `event.data` is shaped from the event you subscribed to, and `ctx.db` is typed from your table schema. Every row your handler writes lands in the materialized table.

```ts
import type { Handler } from "@secondlayer/subgraphs";

// Each matched event becomes one row in the transfers table.
const handler: Handler = async (event, ctx) => {
  const { amount, sender, recipient } = event.data;

  await ctx.db.transfers.insert({
    tx_id: event.txId,
    block_height: event.blockHeight,
    amount,
    sender,
    recipient,
  });
};

export default handler;
```

## Deploy

Deploy from the CLI. The first deploy backfills history from `startBlock`; subsequent blocks stream in automatically.

```bash
sl subgraphs deploy ./subgraph.config.ts
```

## Query your data

Your table is immediately available over REST. Reads are public тАФ cursor-paginate with `_limit` and `_order`.

```bash
curl https://api.secondlayer.tools/api/subgraphs/sbtc-flows/transfers \
  -G -d "_limit=10" -d "_order=desc"
```

## Aggregate without pulling rows

Roll up a table server-side instead of paging every row: `count`, `sum`, `min`, `max`, and distinct counts over the same filters as the list endpoint.

```bash
curl https://api.secondlayer.tools/api/subgraphs/sbtc-flows/transfers/aggregate \
  -G -d "_count=true" -d "_sum=amount" -d "_countDistinct=sender"
```

`_sum`/`_min`/`_max` take numeric columns only. The typed SDK exposes the same as `<table>.aggregate(spec)` тАФ see [SDK aggregates](/docs/sdk#aggregate-subgraph-rows) and the [REST API reference](/docs/rest-api#aggregates) for the full parameter set.

## Index by standard, not by address

A source can target a token standard instead of a fixed contract. `trait` resolves to **every contract Secondlayer classifies as that standard** тАФ including ones deployed after you ship the subgraph тАФ so you index a whole ecosystem with no address list to maintain:

```ts
sources: {
  // every SIP-010 token transfer on-chain
  tokens: { type: "ft_transfer", trait: "sip-010" },
},
```

Supported traits: `sip-009`, `sip-010`, `sip-013`. A reindex backfills each contract's full history from its deploy block, even if it was classified later. See [Contract discovery](/docs/contracts) to query the matching set directly.

## Stream rows in real time

Tail a table over Server-Sent Events as rows are indexed, instead of polling. The endpoint streams from the current tip by default, or pass `?since=<block_height>` to replay from a height and then tail live. Column filters are accepted as query params, and a periodic `ping` event keeps the connection alive.

```bash
curl -N https://api.secondlayer.tools/api/subgraphs/sbtc-flows/transfers/stream \
  -G -d "since=8050000"
```

From the SDK, `subscribe` wraps the stream with the same `where` filters as `findMany` and returns an unsubscribe function:

```ts
const unsubscribe = subgraph.transfers.subscribe(
  (row) => console.log(row),
  { where: { sender: "SP3PE7Q9...X44J" }, since: 8050000, onError: console.error },
);
```

`subscribe` uses the global `EventSource`, so it runs in the browser or **Node тЙе 22** only.

## Use your own database + ORM

Deploy with `--database-url` and your subgraph's rows land in **your** Postgres. Then generate a typed schema for your ORM and query it with full joins and relations:

```bash
sl subgraphs deploy ./subgraph.config.ts --database-url "$DATABASE_URL"
sl subgraphs codegen ./subgraph.config.ts --target prisma -o prisma/schema.prisma
npx prisma generate
```

Prisma and Drizzle have first-class generators; for Kysely, run `kysely-codegen` against your database. Secondlayer owns the table DDL, so treat the tables as read-only and verify with `prisma db pull` (never `migrate`).

Want pushes instead of polling? Bind a [Subscription](/docs/subscriptions) to this table and Secondlayer will POST every matching row change to your webhook тАФ inserts, updates, and deletes, with the verb encoded in the payload `type` (see the [Subscriptions](/docs/subscriptions) page).

### Breaking changes on your own database

For managed subgraphs a breaking schema change (removed table/column, changed column type, or a forced reindex) auto-reindexes by dropping and rebuilding the schema. On a BYO database that drop would destroy your data, so the deploy is **refused** (HTTP `422`) тАФ nothing on your database is touched.

The refusal carries the exact migration plan instead of a generic error. `sl subgraphs deploy` prints the breaking reasons and the `DROP SCHEMA тАж CASCADE` + rebuild DDL to run yourself, then re-deploy:

```bash
тЬЧ Refusing breaking schema change on BYO subgraph (no data dropped).
Breaking changes:
  тЬЧ transfers: removed columns [amount]
To rebuild manually, run on YOUR database:
  DROP SCHEMA IF EXISTS "sg_my_subgraph" CASCADE;
  CREATE SCHEMA "sg_my_subgraph";
  CREATE TABLE тАж
```

The SDK throws a typed `ByoBreakingChangeError` exposing `details.reasons` and `details.plan` (`dropStatement`, `statements`, `grantScript`) for programmatic handling. A destructive `--force` rebuild on your database is **not yet supported** тАФ run the DROP manually.
