Chapter 01

Sources
(what to listen for)

Declare the onchain events you care about. Transfers, contract calls, print events, NFT mints — by kind, by contract, by trait. Subgraphs subscribes on your behalf.

TRACKED events · fts · calls

Sources are the front door. Point them at a contract, a trait, or an event kind — and they start listening from the block you choose.

Most indexers ask you to hand-roll RPC pollers, chain reorgs, retry queues, and cursor storage before you've written a single SELECT. We did that for you. A source is the smallest unit — what kind of event, which contract, since when. Write four lines, get a firehose.

Need more than one source? Declare them. Each one flows into the handler of its own name. Reorgs, pauses, backfills — handled at the boundary. Your code only sees events it can trust.

sources.ts subgraph.config 
// three sources, one declaration
sources: {
  transfer: { type: "stx_transfer" },
  mint:     { type: "nft_mint", contract: "SP...bns-v2" },
  print:    { type: "print_event", topic: "swap" },
}
Chapter 02

Schema
(the Postgres you wanted)

Describe the shape of your tables, with Clarity types that map cleanly to Postgres. Index what you'll query. Nullability where it matters.

TYPES principal · uint · text

Schema is where the chain becomes a database — not a scrape, not a JSON blob, a real set of tables you'd be proud to hand to an analyst.

Every column declares its Clarity type. Principals become Postgres text with checksum validation. Uints become numerics that don't overflow. Buffers become bytea. Indexes are a one-word opt-in.

No more "is that column a bigint or a string this week?". No more silent type coercion burying a bug three months later. The schema is the contract between your handler and your queries — both ends agree, or the deploy refuses.

schema.ts subgraph.config 
schema: {
  transfers: {
    columns: {
      sender:    { type: "principal", indexed: true },
      recipient: { type: "principal", indexed: true },
      amount:    { type: "uint" },
      memo:      { type: "text", nullable: true },
    },
  },
}
Chapter 03

Handlers
(rows from events)

A pure function from event to row. Typed payload in, structured insert out. No RPC, no retries, no cursor bookkeeping.

SHAPE (event, ctx) → ctx.insert

Handlers are the one piece of glue left. Everything else — the replays, the reorgs, the heartbeats — is already solved.

A handler receives a typed event and a context object. You decide what row to write. If you want to read from another table first, use ctx.select. If you want to skip the event, return. That's the whole API surface.

The thing you don't have to do is: touch a node, track a cursor, manage a reconnect, dedupe a replay, or parse a hex buffer by hand. You write business logic. Subgraphs handles the boring, dangerous parts.

handlers.ts subgraph.config 
handlers: {
  transfer: (event, ctx) => {
    ctx.insert("transfers", {
      sender:    event.sender,
      recipient: event.recipient,
      amount:    event.amount,
      memo:      event.memo,
    })
  },
}

Declarative config. Typed everywhere. No framework to learn — if you've written a Next.js config, you've written this.
subgraph.config.ts @secondlayer/subgraphs · v0.12 
import { defineSubgraph } from "@secondlayer/subgraphs"

export default defineSubgraph({
  name: "stx-transfers",
  sources: {
    transfer: { type: "stx_transfer" },
  },
  schema: {
    transfers: {
      columns: {
        sender:    { type: "principal", indexed: true },
        recipient: { type: "principal", indexed: true },
        amount:    { type: "uint" },
        memo:      { type: "text", nullable: true },
      },
    },
  },
  handlers: {
    transfer: (event, ctx) => {
      ctx.insert("transfers", {
        sender: event.sender,
        recipient: event.recipient,
        amount: event.amount,
        memo: event.memo,
      })
    },
  },
})

Then query it
from anywhere.

Subgraphs exposes the same data three ways — pick the one that fits your caller. Every surface is generated from the schema, so your types stay honest across the wire.

SDK TypeScript

Fully-typed client generated from your schema. Autocomplete on column names, returned rows, filters. 

const rows = await sg.transfers
  .where({ sender: addr })
  .orderBy("block", "desc")
  .limit(50)
CLI sl query

Interactive SQL right from your terminal. Pipe results, save snapshots, scaffold dashboards. 

$ sl query stx-transfers \
    "select sender, amount \
     from transfers \
     order by amount desc limit 5"
HTTP REST · SQL

A signed, rate-limited endpoint per subgraph. Fire plain SQL, get JSON rows back. Good enough for cron, good enough for prod. 

POST /v1/stx-transfers/query
{
  "sql": "select count(*) from transfers"
}

Frequently Asked

A few good questions.

If yours isn't here, write to us. We actually reply — usually the same day.

Do I still need to run my own Stacks node? +
No. Subgraphs operates indexers on your behalf. You declare what to listen for; we handle the chain, the reorgs, and the backfill. If you prefer to bring your own node, the Scale plan supports that too.
How is this different from a raw RPC or an archive API? +
RPC gives you a firehose of raw events. Subgraphs gives you SELECT * FROM your_table — shaped, typed, indexed, and replayable. You'd build the same thing yourself, minus a month of work.
What happens on a chain reorg? +
Handlers are idempotent by design. We track block hashes and roll back rows tied to orphaned blocks automatically. Your queries always reflect the canonical chain, not the noisy edge.
Can I query from a smart contract? +
Not directly — Subgraphs is an off-chain read layer. But your dapp, backend, worker, or AI agent can. Three surfaces, same schema: SDK, CLI, HTTP.
